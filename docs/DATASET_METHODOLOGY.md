# WAVE – Dataset Methodology & Project Map

This document explains every decision made in the WAVE ML pipeline, why each
decision was made, and what was changed along the way. Written so that anyone
(including the thesis committee) can reconstruct the reasoning from scratch.
!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!
The original dataset was assembled through a tightly-coupled pipeline that mixed event generation, live weather API calls, and label assignment in a single script. This made the dataset difficult to reproduce and hard to audit. We replaced it with a modular three-step process: (1) generate_foundation.py — creates a balanced 33,000-row interaction grid from 110 real users and 300 synthetic events; (2) a weather enrichment step — attaches synthetic weather conditions per event date/location; (3) a label simulation step — derives attendance labels from user weather-sensitivity profiles and event context. This separation improves reproducibility, makes each assumption explicit, and allows individual steps to be swapped or re-run without affecting others.
!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!!

## 1. What WAVE is trying to do

Build a system that recommends cultural events to a user, taking into account:
1. The user's **personal preferences** (preferred event types, weather tolerances)
2. The **weather on the day** of the event

The research hypothesis: a **contextual model** (with weather features) should
outperform a **baseline model** (without weather features) by a meaningful margin.

## 2. Every file in the project — what it does and why it exists

### Data sources (`data/raw/`)

| File | What it is | Used in pipeline? |
|------|-----------|-------------------|
| `event_attendance.csv` | Kaggle dataset: 200 events, 1000 attendee rows each = 200 000 rows. Columns: event_id, event_name, location (US address with state code), date_time. | Yes – provides event metadata (type, date, location/state) |
| `users_110.csv` | Survey answers from 110 real participants: preferred event types, weather tolerances (1-5), scenario responses (0-3). | Yes – provides real user profiles |
| `weather_cache.csv` | Daily weather fetched from Open-Meteo API for each US state (57 states × 718 days). Generated by `fetch_weather.py`. | Yes – provides real historical weather |
| `cult_pcs_caa_linear.csv` | Eurostat aggregate: % of EU population visiting cinema/theatre by country, year, age, sex. | No – used only as background reading / related work in thesis |
| `cultural_engagement_dataset.csv` | Synthetic-looking interaction table (User_ID, Zone A–D, event type). No link to the 110 survey users or to US weather. | No – out of scope for main pipeline |

### Processing scripts (`src/`)

| Script | What it does | Input → Output |
|--------|-------------|----------------|
| `dataset_pipeline.py` | Cleans Kaggle CSV, extracts temporal features (hour, weekday, month, season), applies Label Encoding for event name and location. Also processes `users_110.csv` into `app_users.csv`. | `event_attendance.csv` → `train_ready.csv` (weather columns empty); `users_110.csv` → `app_users.csv` |
| `generate_weather.py` | **SUPERSEDED – do not run.** Generated *synthetic* weather (temperature, precipitation) correlated with month/season, then flipped some `attended` labels to 0 based on that synthetic weather. This created an artificially easy task. | `train_ready.csv` → overwrites `train_ready.csv` |
| `fetch_weather.py` | Calls Open-Meteo archive API. Fetches **daily** temperature and precipitation for each US state in the dataset. Caches results in `weather_cache.csv`. Joins on (state_code, exact event date) and writes real values into `weather_temp_C` / `weather_precip_mm`. | `train_ready.csv` + API → `train_ready.csv` updated; `weather_cache.csv` |
| `validate_weather.py` | Sanity-check script. Reports: completeness, seasonal averages, attendance rate in good vs bad weather, Pearson correlation, raw sample values. | `train_ready.csv` → printed report |
| `build_interaction_dataset.py` | **NEW (Phase 2).** Builds the user × event × weather interaction table from the 110 real user profiles, 200 Kaggle events, and real daily weather. Each row = one user deciding whether to attend one event on a specific weather day. Attendance label is generated transparently from the user's own survey responses. | `app_users.csv` + `train_ready.csv` + `weather_cache.csv` → `train_ready_interactions.csv` |
| `train_models.py` | Trains 4 models: RF Baseline (no weather), RF Contextual, LGBM Contextual, XGB Contextual. Applies StandardScaler after train/test split (no leakage). Saves models + scaler to `models/`. | `train_ready_interactions.csv` → `models/*.joblib` |
| `evaluate_visuals.py` | Loads saved models, recreates the test split, generates confusion matrices (2×2 grid) and ROC curves. | `train_ready_interactions.csv` + `models/` → `results/` PNG files |
| `evaluate_extended_metrics.py` | Generates metrics comparison table (CSV), grouped bar chart, and Precision-Recall curves. | `train_ready_interactions.csv` + `models/` → `results/` CSV + PNG |
| `cross_validate.py` | 5-fold stratified cross-validation on all 4 models. Reports CV F1 mean ± std and delta vs test F1 to detect overfitting. | `train_ready_interactions.csv` → `results/cv_results.csv` + `results/cv_comparison.png` |
| `debug_pipeline.py` | **Diagnostic only.** Identified the root cause of identical model performance (see Section 4). Can be kept for reproducibility. | `train_ready.csv` + `models/` → `debug-e3238b.log` |
| `build_event_dataset.py` | Intermediate experiment: collapsed 200k rows to 200 event-level rows. Not used in final pipeline; kept for reference. | `train_ready.csv` → `train_ready_event_level.csv` |

### Processed data (`data/processed/`)

| File | What it contains |
|------|-----------------|
| `train_ready.csv` | 200 000 rows from Kaggle with real daily weather joined. Used only for event metadata extraction in Phase 2. |
| `train_ready_interactions.csv` | **Phase 2 training data.** 110 users × 200 events × K weather scenarios. Each row is a unique (user, event, weather day) combination with a transparent attendance label. |
| `app_users.csv` | 110 cleaned survey profiles with weather tolerances, scenario scores, preferred event types. |

All `.joblib` files are regenerated by `train_models.py`. The `scaler.joblib` is

| File | Model |

All generated by `evaluate_visuals.py`, `evaluate_extended_metrics.py`, and

## 3. Why the initial `train_ready.csv` approach was abandoned

### What the Kaggle data actually contains

The Kaggle dataset has **200 events**, each with **1000 attendee rows**. Every
row for the same event has **identical feature values** (same location, same
date, same hour → same weather). Only the `attended` column varies (0 or 1).

This means the model was asked to distinguish attended=1 from attended=0
**using the exact same input vector**. That is mathematically impossible.
No algorithm can do better than random chance on identical inputs with mixed
labels.

### Evidence (from `debug_pipeline.py` / `debug-e3238b.log`)

| Finding | Value |
|---------|-------|
| Unique event_ids | 200 |
| Rows per event | 1000 (exact, every event) |
| Events with BOTH label 0 and label 1 | 200 / 200 (100%) |
| Duplicate feature vectors | 199 800 out of 200 000 |
| Unique feature vectors | 200 |
| Predictions identical: Baseline vs RF Contextual | 100.0% |
| Predictions identical: Baseline vs LGBM | 100.0% |
| Data leakage: same event_id in train AND test | 100% overlap |

### Why "good results" appeared earlier

`generate_weather.py` flipped `attended` labels to 0 **based on the same
synthetic weather it just created**. The model learned: "low temperature =
attended=0" because that rule was baked into the label generation. Remove
the synthetic signal, and the task has no learnable pattern.

## 4. Phase 2: the interaction dataset — what it is and why it is academically valid

### Structure

Each row in `train_ready_interactions.csv` represents:

    one user + one event type + one real weather day → attended (0 or 1)

Key properties:
- **No duplicate feature vectors** (user profiles differ, weather days differ)
- **No data leakage in split** (train/test split is on `user_id`, so no user
  appears in both train and test)
- **Transparent label generation** (formula uses only the user's own survey
  answers + real weather)

### Label generation formula

The attendance probability for row (user u, event e, weather w) is:

```
scenario_score  = user.scenario_{event_type} / 3.0   # 0.0 – 1.0, from survey

weather_penalty = is_outdoor(e) * (
    user.rain_avoid   / 5.0 * clip(precip / 10, 0, 1)     +
    (1 - user.cold_tolerance / 5.0) * clip((5 - temp) / 10, 0, 1) +
    user.heat_sensitivity / 5.0 * clip((temp - 28) / 10, 0, 1)
)

override_reduction = user.override_weather / 5.0 * weather_penalty

final_prob = scenario_score * (1 - weather_penalty + override_reduction)
attended   = Bernoulli(clip(final_prob + noise, 0.05, 0.95))
```

Every variable (`rain_avoid`, `cold_tolerance`, `heat_sensitivity`,
`override_weather`, `scenario_*`) comes directly from the survey. The formula
is written in `src/build_interaction_dataset.py` with `random_state=42` for
full reproducibility.

### Why this is not "circular" or "biased"

The claim in the thesis is NOT "we proved that weather affects attendance in
the real world". The claim is: "our contextual model architecture can capture
the effect of weather on a user's attendance decision, as modelled by their
declared preferences". The dataset operationalises that claim transparently.

This is standard practice in recommender systems research when real interaction
logs are unavailable (see e.g. offline simulation in Jannach et al., 2020).

### Train/test split — GroupShuffleSplit on user_id

Split is performed on `user_id` groups (88 users train / 22 users test). This
means the model is evaluated on **users it has never seen during training**,
which directly mirrors the production use case (new user signs up → gets
recommendations).

### Weather augmentation (K scenarios per user×event pair)

For each (user, event) pair, K=4 real weather days are sampled from
`weather_cache.csv` for the same US state and same season, giving
`110 × 200 × 4 = 88 000` rows. Sampling is stratified across good/neutral/bad
weather conditions to avoid selection bias.

## 5. How to reproduce the full pipeline from scratch

```bash
# 1. Process raw data
python src/dataset_pipeline.py

# 2. Fetch real daily weather (skips states already in cache)
python src/fetch_weather.py

# 3. Build interaction dataset (Phase 2)
python src/build_interaction_dataset.py

# 4. Train all 4 models
python src/train_models.py

# 5. Generate evaluation visuals
python src/evaluate_visuals.py
python src/evaluate_extended_metrics.py

# 6. Cross-validation
python src/cross_validate.py
```

Do NOT run `generate_weather.py` — it overwrites real weather with synthetic
values and re-flips labels artificially.
