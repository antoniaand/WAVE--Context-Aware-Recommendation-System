# WAVE – Dataset Methodology & Project Map

This document explains every decision made in the WAVE ML pipeline, why each decision was made, and what the project currently looks like in its final modular structure. It is written so that anyone (including the thesis committee) can reconstruct the reasoning from scratch.

> [!NOTE]
> The original dataset was assembled through a tightly-coupled pipeline that mixed event generation, live weather API calls, and label assignment in a single script. This made the dataset difficult to reproduce and hard to audit. We replaced it with a fully modular **three-step process**: 
> (1) `generate_foundation.py` — creates a balanced 49,500-row interaction grid from 110 real users and 450 synthetic events across 18 cities; 
> (2) `fetch_weather_api.py` — fetches real historical weather data from the Open-Meteo archive API for every event date and location; 
> (3) `simulate_labels.py` — derives attendance labels transparently from user weather-sensitivity profiles and event context via a high-fidelity behavioural simulation.
> This approach guarantees that data creation is strictly reproducible, and the model's environment can be audited securely.

---

## 1. What WAVE is trying to do

Build a system that recommends cultural and entertainment events to a user, taking into account:
1. The user's **personal preferences** (preferred event types, scenario scores)
2. The **weather on the day** of the event (and the user's sensitivity to weather conditions like heat, cold, and rain)

The research hypothesis: a **contextual model** (with real weather features + user sensitivity features) should outperform a **baseline model** (without weather-related features) by a meaningful margin, particularly for outdoor events under extreme conditions.

## 2. Every file in the project — what it does and why it exists

### Data sources (`data/raw/`)

| File | What it is | Used in pipeline? |
|------|-----------|-------------------|
| `users_110.csv` | Survey answers from 110 real participants: preferred event types, weather tolerances (1-5 scale), and scenario responses (0-3 scale). | Yes – provides the base set of real user profiles used to seed the simulator. |
| `weather_archive_cache.csv` | Hourly historical weather fetched from the Open-Meteo API for 18 specific cities. Generated by `fetch_weather_api.py`. | Yes – caches real API results so re-running the pipeline is fast and doesn't hit the API rate limits. |

### Processing scripts (`src/`)

The pipeline has been logically modularized into distinct directories.

**`src/data/` (Data Generation Pipeline)**
| Script | What it does | Input → Output |
|--------|-------------|----------------|
| `generate_foundation.py` | Generates 450 synthetic events across 4 climate zones, and a full Cartesian product with 110 users, creating 49,500 rows. No weather yet. | `users_110.csv` → `interaction_foundation.csv` |
| `fetch_weather_api.py` | Scrapes historical weather data from Open-Meteo for the 18 unique cities and attaches `temperature_2m`, `precipitation`, `windspeed`, etc., matching the exact event date and hour. | `interaction_foundation.csv` + API → `interaction_with_weather.csv` |
| `simulate_labels.py` | Builds a `final_prob` threshold and `attended` label by taking user weather sensitivities, multiplying by real weather conditions, and generating a 50/50 balanced dataset. | `interaction_with_weather.csv` → `train_ready_interactions.csv` |

**`src/modeling/` (Machine Learning Training)**
| Script | What it does |
|--------|-------------|
| `train_models.py` | Trains 4 core classifiers (RF Baseline, RF Baseline Strict, RF Contextual, LGBM Contextual, XGB Contextual) using strict data split boundaries and a unified `StandardScaler`. Outputs `.joblib` models to `/models`. |
| `tune_models_OPTUNA.py` | Uses the Optuna framework to run Hyperparameter searches. Target used prior to baking into pipeline. |
| `tune_models_GRIDSearch.py` | Simple GridSearchCV for initial model capability tests. |
| `cross_validate_overfitting.py` | Runs 5-Fold Stratified Cross-Validation on all four models to diagnose overfitting before using the unseen hold-out split. Outputs `cv_results.csv`. |

**`src/evaluation/` (Post-Training Evaluation)**
| Script | What it does |
|--------|-------------|
| `eval_common.py` | Pre-processing boilerplate loader used uniformly by all evaluation scripts. |
| `evaluate_visuals.py` | Generates plots: Confusion Arrays, ROC curves, Subgroup slices, permutation analysis. Outputs to `results/visual_performance/` and `results/visuals_subgroups/`. |
| `evaluate_extended_metrics.py` | Calculates tabular performance, specifically zooming in entirely on "Extreme Weather Subgroup" rows. Computes Accuracy, F1 gaps and the Relative Error Reduction scores. Outputs CSVs to `results/metrics/`. |
| `scenario_validation.py` | Calibrates the `simulate_labels.py` probability mechanism directly against real hypothetical answers reported by the dataset subjects to ensure survey mapping is realistic. |
| `test_weather_signal.py` | Dedicated isolated sandbox to confirm specifically if the baseline algorithm exploits any un-scrubbed feature correlations to implicitly "learn" weather patterns even when technically "blinded". |

**`src/xAI_SHAP/` (Explainable AI)**
| Script | What it does |
|--------|-------------|
| `explain_models.py` | Calculates TreeSHAP explanations specifically on the deployed models and renders localized/global interpretation visuals to `results/explainability/`. |

### Processed data (`data/processed/`)

| File | What it contains |
|------|-----------------|
| `app_users.csv` | Cleaned up data format for the 110 survey users, stripped of email metadata but containing their core behavioral features. |
| `interaction_foundation.csv` | Output from step 1. Exactly 49,500 target events. |
| `interaction_with_weather.csv` | Output from step 2. Features appended from the Open-Meteo API. |
| `train_ready_interactions.csv` | Output from step 3. Features ready and cleaned for the Data Loaders. |

### Models (`models/`)

All `.joblib` files are regenerated by `train_models.py`. The `scaler.joblib` is fitted on training data only (strictly enforcing no test-data leakage).

| File | Model |
|------|-------|
| `baseline_rf.joblib` | RandomForest without weather features |
| `baseline_strict_rf.joblib` | RandomForest baseline that strictly drops location, climate_zone, and event_month features avoiding proxy data-leaks. |
| `contextual_rf.joblib` | RandomForest with explicit real weather features + profile features. |
| `lgbm_contextual.joblib` | LightGBM with weather features |
| `xgb_contextual.joblib` | XGBoost with weather features |
| `scaler.joblib` | StandardScaler fitted on X_train fold only |

## 3. The dataset rationale

### Structure

Each row in `train_ready_interactions.csv` represents:

    one user + one event type + one real historical weather day → attended (0 or 1)

Key property guarantees:
- **No duplicate feature vectors** (User profiles differ dynamically and historical locations reflect varying micro-climates).
- **No data leakage in split** (Train/test split is grouped on `user_id`, meaning no user profile appears in both train and test partitions).
- **Transparent label generation** (Uses solely the user's recorded preferences interacting with realistic Open-Meteo historical constraints).

### Label generation formula (implemented in `simulate_labels.py`)

The attendance probability for a row is largely an aggregation of a base desire and a weather adjustment penalty.

A `weather_penalty` relies fundamentally on evaluating:
```python
DI = 1.8 * T - 0.55 * (1.8 * T - 26) * (1 - F) - 3.2 * np.sqrt(Ws) + 32
```
Building a Discomfort Index (Zhang et al. 2020) out of the raw factors. Heavy Rain or deviations from Ideal bounds reduce the continuous probabilty scale `final_prob`. An `attends` label is simply applied along a unified median decision boundary ensuring deterministic fairness.

This mechanism enforces a realistic model: users can dynamically overcome bad weather if their specific `override_weather` or base scenario inclination is sufficiently motivated, mimicking typical consumer logic.

### Why this is academically valid

The research objective of WAVE is NOT "we proved that bad weather affects real-world attendance." The claim is: **"Our explicit contextual model architecture can effectively capture ad-hoc data signals representing weather variations—dynamically evaluating user tolerances—and will materially outperform generalized rigid baselines at predicting those decisions."** The structured offline simulator dataset operationalises that very claim transparently and verifiably, mirroring typical cold-start offline simulations in recommender system methodology literature.

## 4. How to reproduce the full pipeline from scratch

Every step is idempotent and explicitly scoped. The dataset will rebuild exactly from the seeds.

```bash
# 1. Step 1 -- Generate foundation Cartesian dataset
python src/data/generate_foundation.py

# 2. Step 2 -- Fetch real hourly weather from Open-Meteo
python src/data/fetch_weather_api.py

# 3. Step 3 -- Simulate labels
python src/data/simulate_labels.py

# 4. Step 4 -- Train all the models
python src/modeling/train_models.py

# 5. Step 5 -- Evaluate performance and visual artifacts
python src/evaluation/evaluate_visuals.py

# 6. Step 6 -- Generate granular metrics
python src/evaluation/evaluate_extended_metrics.py
```
